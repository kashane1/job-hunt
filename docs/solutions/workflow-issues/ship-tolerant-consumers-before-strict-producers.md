# Ship tolerant consumers before strict producers when introducing a new invariant across phases

## Problem

Batch 4's plan introduces a per-surface playbook contract: each playbook declares a `checkpoint_sequence` in YAML frontmatter, and `record_attempt` validates that every reported checkpoint is a member of that sequence (`ApplicationError(plan_schema_invalid)` otherwise).

The producer of `checkpoint_sequence` is Phase 5 (per-surface playbooks).
The consumer of `checkpoint_sequence` is Phase 4 (`record_attempt`).

Three plausible orderings, each broken in a different way:

1. **Strict consumer first (Phase 4) → producer (Phase 5)**: every test that exercises `record_attempt` raises `plan_schema_invalid` because no playbook has the frontmatter yet. The repo is broken between Phase 4 and Phase 5 landing.

2. **Producer first (Phase 5 ships ahead of Phase 4)**: the playbooks have frontmatter that nothing reads. Reviewers can't tell whether the contract is real. Phase 4 still has to introduce the reader; bundling them would just collapse the phase boundary the planner chose for shippability reasons.

3. **Bundle Phase 4 + Phase 5 into one PR**: defeats the whole point of phase boundaries. A 5-day Phase 4 + 4-day Phase 5 becomes a 9-day PR with two test surfaces. The plan was specifically split so Phase 4 could land and be reviewed independently.

## Solution

**Ship the consumer in tolerant mode first; the producer adds the data; a third change promotes the consumer to strict.**

Three concrete steps from Batch 4:

1. **Phase 4 — tolerant consumer.** `playbooks.load_checkpoint_dag(playbook_path)` returns `[]` when the file or frontmatter is missing. `record_attempt` only enforces the DAG when the loader returns a non-empty list:

   ```python
   # src/job_hunt/playbooks.py — Phase 4
   def load_checkpoint_dag(playbook_path: str) -> list[str]:
       if not playbook_path:
           return []
       path = Path(playbook_path)
       if not path.is_absolute():
           path = repo_root() / playbook_path
       if not path.is_file():
           return []
       text = path.read_text(encoding="utf-8")
       frontmatter, _ = parse_frontmatter(text)
       sequence = frontmatter.get("checkpoint_sequence", [])
       if not isinstance(sequence, list):
           return []
       return [str(s) for s in sequence]
   ```

   ```python
   # src/job_hunt/application.py — Phase 4 record_attempt
   checkpoint_sequence = load_checkpoint_dag(plan.get("playbook_path", ""))
   checkpoint = attempt_payload.get("checkpoint")
   if checkpoint_sequence and checkpoint not in checkpoint_sequence:
       raise ApplicationError(
           f"Checkpoint {checkpoint!r} not in declared sequence {checkpoint_sequence}",
           error_code="plan_schema_invalid",
           remediation="Align the playbook's checkpoint_sequence with the attempt checkpoint.",
       )
   ```

2. **Phase 5 — producer adds the data.** Each per-surface playbook's frontmatter declares the sequence:

   ```yaml
   # playbooks/application/indeed-easy-apply.md
   ---
   playbook_id: indeed-easy-apply
   surface: indeed_easy_apply
   checkpoint_sequence:
     - preflight_done
     - form_opened
     - fields_filled
     - ready_to_submit
     - submitted
     - confirmation_captured
   ---
   ```

   At this point Phase 4's `record_attempt` is automatically enforcing the DAG for every playbook that has frontmatter. Playbooks without frontmatter still pass (tolerant). Phase 5's tests assert both directions — legal sequence passes, fabricated checkpoint rejected.

3. **Phase 9 — promote tolerance to strict.** `check-integrity` adds `playbook_missing_checkpoint_sequence`: any per-surface playbook referenced by `plan.surface` but missing the frontmatter is now a hard-fail. The tolerance gate moves from "consumer ignores missing data" to "infrastructure forbids missing data."

   ```python
   # src/job_hunt/tracking.py — Phase 9 check_integrity
   playbook_dir = data_root.parent / "playbooks" / "application"
   if playbook_dir.is_dir():
       from .playbooks import load_checkpoint_dag
       for pb in playbook_dir.glob("*.md"):
           if pb.stem == "generic-application":
               continue  # router; no checkpoint sequence by design
           seq = load_checkpoint_dag(str(pb.relative_to(data_root.parent)))
           if not seq:
               playbook_missing_checkpoint_sequence.append({"path": str(pb)})
   ```

The same pattern repeated for `origin_allowlist` (Phase 5 frontmatter, Phase 4 `load_origin_allowlist` tolerant reader, Phase 9 forbids absence on per-surface playbooks).

## Why This Is Better Than The Alternatives

| Approach | Pro | Con |
|----------|-----|-----|
| Bundle producer + consumer in one phase | Invariant active immediately | Defeats phase boundaries; review blast radius doubles |
| Strict consumer first | Invariant immediately enforced | Repo broken between consumer and producer; tests fail mid-stream |
| Producer first | Data exists waiting for reader | Phase boundary intact, but reviewers can't tell the contract is real until phase N+2 |
| **Tolerant consumer → producer → strict promotion** | **Phase boundaries intact; each phase independently reviewable; each phase ships green; invariant becomes hard once everyone's compliant** | Three phases instead of two; risk of forgetting to promote (mitigated by check-integrity) |

The promotion step is what separates "tolerant by design" from "tolerant by accident." Without Phase 9's `check-integrity` extension, the tolerant loader could silently allow regressions forever — a new playbook author could omit frontmatter and nothing would notice.

## Where Else This Pattern Applies

Same shape, different domains:

- **Schema field rollout**: ship the writer in "set the field if known" mode, ship the reader in "use field if present" mode, then a third change makes the field required and back-fills any holdouts. Fundamentally the same producer/consumer split.
- **Feature flags**: ship the gated code (consumer) reading the flag (producer) defaulting to off, ship the flag turn-on, ship the un-gating. The flag IS the producer, the gated code IS the consumer.
- **Config-driven validation**: ship a validator that no-ops when the config key is absent, ship the config, ship the strict promotion that errors on missing key. (`config/domain-allowlist.yaml` followed an inversion of this — the loader returns an empty set when the file is missing, preserving v1 "everything hard-fails" rather than "everything passes." The pattern flexes either direction.)
- **Migration sequencing**: a column rename done as add-new + dual-write + read-from-new + drop-old is the same idea — never have a moment where consumers expect data that producers aren't yet writing.

## Prevention / Best Practices

When planning a multi-phase change that introduces a new invariant:

1. **Identify the producer/consumer split early.** During the plan-deepening pass, list every "Phase N writes data Phase M reads." If N > M (consumer ships before producer), the consumer needs tolerant mode.
2. **Make the tolerance explicit in the consumer's docstring.** Write: *"Phase 4 ships this loader tolerant of missing frontmatter. Phase 5 adds the frontmatter. Phase 9 promotes check-integrity to hard-fail on missing frontmatter."* That comment is what saves the next reader from thinking the tolerant return is a bug.
3. **Always add a promotion step.** A tolerant consumer without a strict-promotion follow-up is just "permanently silent." The promotion goes in a hardening phase (typically the last one in the batch). It's not optional — schedule it explicitly.
4. **Use `check-integrity` as the strict-mode enforcement point.** Adding a check there is much cheaper than promoting the runtime path to strict, because `check-integrity` is the natural place to surface "data the system would tolerate but the team should not."
5. **Test both directions in the producer phase.** When Phase 5 ships the producer, its tests assert (a) the consumer accepts a record with the new shape, AND (b) the consumer rejects an obviously-bad record. This catches the case where the consumer's tolerance was actually a bug.

## Tests / Verification

In Batch 4's case, the verification chain looked like:

- Phase 4 tests: `record_attempt` accepts any checkpoint when no playbook frontmatter exists (tolerant mode confirmed).
- Phase 5 tests (`tests/test_playbooks.py`): every playbook has `checkpoint_sequence` with `preflight_done` + `ready_to_submit`; `record_attempt` rejects a fabricated checkpoint when the playbook has frontmatter; `record_attempt` accepts a checkpoint that's first in the sequence.
- Phase 9 tests (`tests/test_phase9_integrity.py`): `check_integrity` flags any per-surface playbook missing `checkpoint_sequence` (would fire if a future playbook author forgets frontmatter).

Total cost: ~4 lines of code in the consumer (the `if checkpoint_sequence:` guard), plus the promotion check in `check-integrity`. Cost is dominated by writing the docstring and the tests, not the runtime logic.

## What This Compounds

The first time this pattern arose (Batch 4 playbooks), the deepening review caught the cyclic-prerequisite as a P2 finding (v3 item #22). Without the catch, Phase 4's tests would have failed against Phase 5's still-absent playbook frontmatter, and the natural fix would have been to bundle Phase 4 + Phase 5 — collapsing the phase boundary the planner chose for review tractability.

Documenting the tolerant-→-strict pattern means:

- Future multi-phase plans in this repo can adopt the pattern without re-deriving it. Look for "Phase N reads data Phase M writes" during the deepening review and apply the three-step recipe.
- The pattern generalizes to any cross-team or cross-service contract introduction — feature flags, schema migrations, config validators. The mental model "consumer tolerant, producer adds, infrastructure promotes" is portable.
- The hardest discipline is the promotion step. Recording it explicitly here (as a required phase artifact, not an aspirational one) keeps the tolerance from rotting into permanent silent failure.
